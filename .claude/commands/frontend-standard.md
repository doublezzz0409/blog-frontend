### 第一阶段：数据与接口清算（前后端共同契约）

步骤1：从用户提供的 UI 设计稿和需求描述中，提取所有数据实体（例如用户、列表项、表单字段），为每个实体定义 TypeScript interface。明确每个字段的名称、类型（string / number / boolean / 枚举值）、是否可选。将所有 interface 写入 `types.ts` 文件。你的输出必须是完整的 `types.ts` 内容，并在末尾注明"✅步骤1 完成"。

**步骤1 补充约束（字段格式精确性强制规则）：**

a) **类型精度**
- 每个字段的类型必须是精确的，禁止使用 `any`、`unknown`。

b) **ID 字段**
- 必须明确是 number 还是 string。
- 如果是 string，需进一步明确格式规则（UUID / 自增数字字符串 / 带前缀如 `"art-{uuid}"` / 其他）。
- 所有实体的 ID 格式必须在本步骤确定并统一，形成项目的"ID 命名规范"。同一实体家族（如 `Article` 和 `ArticleDetail`）的 ID 格式必须一致。
- 示例规范：文章=`"art-{id}"`，分类=`"cat-{id}"`，标签=`"tag-{id}"`，用户=`"user-{id}"`，评论=`"cmt-{id}"`。

c) **可空字段**
- 必须明确是 `null` 还是空字符串 `""` 或空数组 `[]`，禁止"可为空字符串或 null"的双重可能。
- **统一原则：可空的字符串字段默认使用空字符串 `""`，可空的数组字段默认使用空数组 `[]`，可空的对象字段使用 `null`。项目全领域保持一致。**

d) **日期/时间字段**
- 必须指定格式（ISO 8601 UTC 含时区 `Z` / Unix timestamp（秒/毫秒） / 仅日期 `YYYY-MM-DD`）。
- 默认标准：所有日期时间字段统一使用 ISO 8601 UTC 格式，如 `"2026-04-30T12:00:00Z"`。

e) **枚举型字段**
- 必须列出全部可能的值（如 `status: "draft" | "published" | "archived"`），并将枚举字符串写入 JSDoc 注释。

f) **字符类字段的字符集约束**
- 对于 slug、username 等 URL 或标识类字段，必须明确允许的字符集（如"仅小写字母、数字、连字符 `-`，以字母开头，长度 1-200"）。
- 禁止使用"URL 友好标识"等仅人类可读的模糊描述。

g) **数值字段的范围约束**
- 对于可能影响性能的数值字段（如分页大小），必须明确上下限（如"正整数，范围 1-100"）。禁止无上下限。

h) **待确认项处理**
- 任何无法在当下确定的格式，**必须在该字段的 JSDoc 注释中标记 `// TODO: 待与后端确认`**，并在此步骤末尾汇总所有待确认项，逐条列出字段路径和待确认内容。
- 禁止写"格式不限""任意""视情况而定"等表述来替代待确认标记。

步骤2：基于用户需求和 `types.ts`，列出页面涉及的所有 API 端点（HTTP 方法、路径、功能描述）。为每个端点定义请求体/查询参数的类型，以及完整的响应体类型（必须包含 `code`, `data`, `message` 外壳，`data` 必须引用 `types.ts` 中的类型）。将所有 API 类型定义写入 `api-contract.ts`。输出 `api-contract.ts` 完整内容，并注明"✅步骤2 完成"。

**步骤2 补充约束：**
- 每个 API 的请求体/响应体类型必须完全引用步骤1中定义的 interface，不得在此步骤重新定义字段格式。
- 如果步骤1存在"待确认"项，本步骤生成的 api-contract 中对应的类型定义必须保留 `// TODO` 标记，不得自行填补。
- 对于写操作接口（POST/PUT/DELETE），需明确是全量更新还是部分更新。默认 RESTful 约定：`PUT` = 全量更新（请求体包含全量字段），`PATCH` = 部分更新。若使用 `PUT` 做部分更新，必须在注释中说明。

步骤3：基于 `api-contract.ts`，为每个 API 编写封装好的服务函数（例如 `getUserProfile()`, `updateNickname()`），函数必须明确参数类型和返回类型，不允许直接在组件中调用 `fetch`/`axios`。将服务函数写入 `services.ts`。输出 `services.ts` 完整内容，并注明"✅步骤3 完成"。

**步骤3 补充约束：**
- 服务函数的参数和返回类型必须与 `api-contract.ts` 严格一致，不允许放宽或收紧。
- 服务函数必须统一处理 `code !== 200` 的情况，将其作为错误抛出（抛出 `message` 字段内容）。
- 服务函数不得修改请求体或响应体的字段名或格式。
