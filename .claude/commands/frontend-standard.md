## 你的指令（全栈开发全流程强制执行）

### 第一阶段：数据与接口清算（前后端共同契约）
步骤1：从用户提供的 UI 设计稿和需求描述中，提取所有数据实体（例如用户、列表项、表单字段），为每个实体定义 TypeScript interface。明确每个字段的名称、类型（string / number / boolean / 枚举值）、是否可选。将所有 interface 写入 `types.ts` 文件。你的输出必须是完整的 `types.ts` 内容，并在末尾注明“✅步骤1 完成”。

步骤2：基于用户需求和 `types.ts`，列出页面涉及的所有 API 端点（HTTP 方法、路径、功能描述）。为每个端点定义请求体/查询参数的类型，以及完整的响应体类型（必须包含 `code`, `data`, `message` 外壳，`data` 必须引用 `types.ts` 中的类型）。将所有 API 类型定义写入 `api-contract.ts`。输出 `api-contract.ts` 完整内容，并注明“✅步骤2 完成”。

步骤3：基于 `api-contract.ts`，为每个 API 编写封装好的服务函数（例如 `getUserProfile()`, `updateNickname()`），函数必须明确参数类型和返回类型，不允许直接在组件中调用 `fetch`/`axios`。将服务函数写入 `services.ts`。输出 `services.ts` 完整内容，并注明“✅步骤3 完成”。

### 第二阶段：Mock 环境搭建
步骤4：安装 MSW 依赖（`npm install msw --save-dev`）并初始化（`npx msw init public/ --save`，若目录不同须根据项目调整）。然后创建 `mocks/handlers.ts`，为 `api-contract.ts` 中列出的**每一个** API 编写：
- 至少一个成功响应的处理器（返回数据严格匹配 api-contract 类型）
- 所有写操作（POST/PUT/DELETE）分别编写至少一个典型失败响应（400 业务错误 + 401 未授权 + 500 服务器错误），每个失败响应必须标注其业务含义

处理器返回的数据结构必须严格匹配 `api-contract.ts` 中的类型。输出 `mocks/handlers.ts` 完整内容，并注明“✅步骤4 完成”。

步骤5：创建 `mocks/browser.ts`，使用 `setupWorker` 导入 handlers 并启动。然后在应用入口文件（如 `main.tsx`）中引入并调用 `worker.start()`，仅在开发模式下启用。输出 `mocks/browser.ts` 内容及相关入口文件的修改说明，并确认 Mock 已启用。标注“✅步骤5 完成”。

### 第三阶段：组件原子化开发
步骤6：基于 UI 设计稿，画出组件树（父组件与子组件的层级关系），列出每个组件的名称、职责（只允许 1-2 个职责），并明确哪些组件有内部状态、哪些完全受控。输出组件树（文字描述或简图）和职责说明，标注“✅步骤6 完成”。

步骤7：为步骤6中列出的每一个组件定义 Props 接口（输入只通过 Props）。若组件需要异步数据，则定义其内部状态枚举：`loading | empty | error | ready`。先输出所有组件的签名草案（如 `function NicknameDisplay(props: { nickname: string; onEdit: () => void })`），等待用户确认名称和职责后再继续。确认后立即单个逐一实现，不能跳步。每实现完一个组件，输出该组件的完整代码，并标注“✅步骤7-X 完成”。所有组件实现完毕后，标注“✅步骤7 完成”。

步骤8：逐个检查所有含异步数据组件的四种状态实现：
- Loading 态（骨架屏或转圈，不得完全空白）
- Empty 态（占位提示，不得消失或报错）
- Error 态（错误信息+重试按钮，不得吞掉错误）
- Ready 态（正常数据展示）

若缺失任何一种状态，立即补齐。每补齐一个组件的状态，输出该组件的更新代码，并标注“✅步骤8-X 完成”。全部补完后，标注“✅步骤8 完成”。

### 第四阶段：路由与导航契约清算（🔴 2026年新增——弥补天生盲区）
> **设计意图：你在组件里写的每一个 `<Link>`、`useNavigate()`、`window.location` 都是一份“路由需求”。你必须证明这些需求已被路由表满足。这不是可选的检查，这是契约。**

步骤9：扫描项目中所有组件文件（`.tsx`、`.jsx`），提取所有路由跳转点：
- `<Link to="...">` 或 `<NavLink to="...">`
- `navigate("...")` 或 `history.push("...")`
- `window.location.href = "..."` 或 `router.push("...")`

整理成**路由需求清单**，每一条包含：跳转目标路径、所在文件及行号、跳转上下文（点击按钮/自动跳转/条件跳转等）。输出完整清单，标注“✅步骤9 完成”。

步骤10：读取路由注册文件（通常为 `App.tsx` 或独立的路由配置文件），提取所有已注册路由。将步骤9的路由需求清单与已注册路由做**全量对照**：
- 需求清单中的路径必须能在路由表中精确匹配或模式匹配（如 `/categories/:id` 匹配 `/categories/123`）
- 任何**未匹配的路径**必须立即处理，选项包括：
  a) 在路由表中注册对应路由
  b) 将链接改为已存在的路由（如改为带 query 参数的首页路径）
  c) 添加重定向规则

处理完毕后，输出对照表和处理结果，标注“✅步骤10 完成”。

### 第五阶段：组件交叉校验（防止割裂）
步骤11：执行以下交叉检查并输出结果：
1. 所有组件中 import 的类型、函数是否真实存在于 `types.ts`、`services.ts` 或对应组件文件中。
2. 所有组件中调用的 API 函数是否真实存在于 `services.ts`。
3. 所有 services 函数中请求的 API 端点和请求/响应结构是否与 `api-contract.ts` 一致。
4. 路由需求清单中所有路径对应的页面组件是否真实存在且被正确导出。

若发现不一致，必须在继续前修复。全部通过后标注“✅步骤11 完成”。

### 第六阶段：双模冒烟验证
> **设计意图：测试必须包含“正向验证”和“逆向发现”两个动作。只测已注册路由是正向思维——你必须同时证明“不存在链接指向未注册路由”。**

步骤12：使用 Playwright（或用户指定的 E2E 框架）编写自动化测试脚本，分两部分：
**Part A：可达性测试（正向验证）**
- 遍历步骤10中确认的每一个已注册页面路由，访问并断言页面不为空白（至少有一个可见元素或特定标识文本）。
**Part B：全量链接扫描（逆向发现）**
- 基于步骤9的路由需求清单，逐一访问存在链接的源页面，点击或触发所有链接，断言跳转后页面不为空白、无 404 文本、无控制台路由错误。

输出测试脚本内容，标注“✅步骤12 完成”。

步骤13：启动项目（Mock 模式），运行步骤12的测试脚本。测试必须 100% 通过。若失败，必须提供：
- Playwright 详细错误日志
- 网络请求快照（若有 API 调用失败）
- 失败的链接路径及所在文件

然后最小化修复，重跑直到通过。通过后标注“✅步骤13 完成”。

### 第七阶段：后端交付文档
步骤14：Mock 环境验证通过后，生成 `mock-api-doc.md`，面向后端开发人员，包含：
- 所有 API 端点列表（方法、Path、Content-Type、Headers 示例）
- 每个端点成功和典型失败场景的请求体/响应体示例
- 每个响应字段的含义说明
- Mock 中已模拟的成功/失败分支及业务含义
- 一段“前后端契约对齐说明”，强调前端严格依此文档开发，后端实际接口须与此兼容

输出文档内容并标注“✅步骤14 完成”。
